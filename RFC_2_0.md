# RFC: kornia.augmentations 2.0 — fast-default architecture redesign

**Status:** Draft
**Date:** 2026-04-28
**Author:** kornia maintainers
**Supersedes:** Parts of RFC 1.0 (the funnel) — specifically items deferred as kornia 2.0 work.
**Tracking issues:** torchgeo/torchgeo#3108, roboflow/rf-detr#874

## TL;DR

The kornia 1.x augmentation framework pays for "every feature available on every call". On Jetson Orin GPU this costs 6-80× per-op vs torchvision.v2 even after aggressive eager-mode optimization, because the framework eagerly constructs `_params`, `transform_matrix`, `batch_prob`, and routes through a 5-level inheritance hierarchy regardless of what the caller actually needs.

This RFC proposes a redesigned architecture that makes every kornia-unique capability **opt-in via tensor types or explicit flags**, with the *default* path being torchvision-fast. After redesign, kornia in default mode is within 1.3-2.5× of torchvision per-op, while the kornia-unique capabilities (per-sample params, multi-target dispatch, transform_matrix tracking, replay, inverse, 200+ ops, 3D, video, geometric richness) remain available for callers who need them.

## 1. Goals

- **G1.** Match torchvision.v2 within 1.3-2.5× per-op on default (batch-shared) mode.
- **G2.** DETR pipeline drops from 58 ms to ~15 ms (~4× speedup).
- **G3.** Preserve all kornia-unique capabilities: per-sample params (opt-in), transform_matrix tracking (lazy), replay (opt-in), inverse (mixin), multi-target dispatch (type-driven), 200+ ops, 3D, video, geometric richness.
- **G4.** Backwards compatibility via two-namespace coexistence — `kornia.augmentation` (1.x) keeps working unchanged for one major version while `kornia.augmentations` (2.0) ships the new fast path.
- **G5.** Code reduction: ~40-50% fewer lines in the augmentation package.
- **G6.** Empirically grounded — every architectural choice is justified by torch.profiler bottleneck data.

## 2. Non-goals

- **NG1.** Differentiability through augmentations is **out of scope**. All augmentation transforms run under `@torch.no_grad()`. Users needing gradient-preserving normalization for in-model preprocessing import `kornia.enhance.normalize` directly. The `NormalizeWithGrad` carve-out from RFC 1.0 is removed in 2.0.
- **NG2.** TorchScript scripting compatibility for augmentation transforms — Pydantic in `__init__` and the type-dispatch registry don't script. Users needing scripted preprocessing use the deterministic functional API.
- **NG3.** Closing the per-sample-mode gap to torchvision — kornia's per-sample mode does structurally more work (B different transforms vs 1 broadcast). Per-sample is the kornia value-add, priced at ~2-3× of torchvision same-mode.
- **NG4.** Replacing `F.grid_sample` — the kernel floor for geometric ops on consumer GPUs. Custom warp kernels are out of scope; kornia uses what PyTorch ships.
- **NG5.** `kornia.geometry`, `kornia.color`, `kornia.feature`, etc. — no changes outside `kornia.augmentation`/`kornia.augmentations`.

## 3. Background — what the profile data shows

torch.profiler runs on Jetson Orin GPU with the v6 patches active (B=8, 512×512, fp32) reveal a clear pattern:

| Op | events/iter | tv events/iter | k vs tv ratio |
|---|---:|---:|---:|
| CenterCrop | 40 | 5 | 80× slower |
| HFlip | 22 | 17 | 5.5× slower |
| Grayscale | 62 | 28 | 11× slower |
| Affine | 466 | 97 | 7.3× slower |
| ColorJitter | 441 | 239 | 2.3× slower |
| CutMix | 866 | 81 | 43× slower |

Top non-kernel events on every kornia op are `aten::full`, `aten::empty`, `aten::to`, `aten::lift_fresh`, `aten::eye` — all generated by per-call construction of:
- `batch_prob` tensor (B,) bool — even when `p=1.0` (constant True; pure waste)
- `forward_input_shape` tensor (4,) long — used by very few transforms
- `transform_matrix` (B,3,3) — built for every rigid transform regardless of whether anyone reads it
- `_params` dict — constructed for every call, used only when `aug.inverse()` or `params=` replay is called

Conclusion: **the per-op gap is dominated by framework overhead, not kernel cost.** Fixing this requires architectural changes, not inline optimization.

## 4. Architecture

### 4.1 The new base class

```python
# kornia/augmentations2/base.py
from typing import Any, Callable
import torch
from torch import nn

_KERNEL_REGISTRY: dict[tuple[type, type], Callable] = {}

def register_kernel(transform_cls: type, tensor_cls: type) -> Callable:
    """Register a per-(transform, tensor-type) kernel for type-dispatch."""
    def deco(fn: Callable) -> Callable:
        _KERNEL_REGISTRY[(transform_cls, tensor_cls)] = fn
        return fn
    return deco


class Transform(nn.Module):
    """Minimal augmentation base. Always no_grad. ~50 lines.

    Subclass directly for deterministic image-only ops (Normalize, CenterCrop).
    Add mixins for randomness (_RandomMixin), rigid-affine matrix tracking
    (_RigidMixin), inverse support (_InvertibleMixin), or multi-target dispatch
    (_MultiTargetMixin).
    """

    def __init__(self) -> None:
        super().__init__()
        self._last_input_shape: torch.Size | None = None
        self._last_params: dict | None = None
        self._cached_matrix: torch.Tensor | None = None

    def _sample_params(self, shape: torch.Size, generator=None) -> dict:
        """Override if random. Default: empty (deterministic ops)."""
        return {}

    def _apply(self, x: torch.Tensor, params: dict) -> torch.Tensor:
        """Required override. Pure tensor in, pure tensor out."""
        raise NotImplementedError

    def _dispatch(self, inputs: tuple, params: dict):
        """Single-target by default. _MultiTargetMixin uses the kernel registry."""
        if len(inputs) == 1:
            return self._apply(inputs[0], params)
        outputs = []
        for inp in inputs:
            kernel = _KERNEL_REGISTRY.get((type(self), type(inp)))
            outputs.append(kernel(self, inp, params) if kernel else inp)
        return tuple(outputs)

    @torch.no_grad()
    def forward(self, *inputs, generator=None, params=None, return_params=False):
        if not inputs:
            raise ValueError("at least one input required")
        x = inputs[0]
        if params is None:
            params = self._sample_params(x.shape, generator)
        output = self._dispatch(inputs, params)
        # Lazy metadata
        self._last_input_shape = x.shape
        self._last_params = params if return_params else None
        self._cached_matrix = None
        if return_params:
            return output, params
        return output

    @property
    def transform_matrix(self) -> torch.Tensor | None:
        return None  # _RigidMixin overrides
```

**~50 lines.** Compared to current `_BasicAugmentationBase` which spans multiple thousand lines across 5 files and 5 inheritance levels.

### 4.2 Mixins

Composition-based, opt-in:

```python
class _RandomMixin:
    """Marker for stochastic transforms. No behavior at this layer."""

class _RigidMixin:
    """Adds lazy transform_matrix support."""
    def _compute_matrix(self, params: dict, shape: torch.Size) -> torch.Tensor:
        raise NotImplementedError
    @property
    def transform_matrix(self) -> torch.Tensor | None:
        if self._last_input_shape is None:
            return None
        if self._cached_matrix is None:
            params = self._last_params or self._sample_params(self._last_input_shape)
            self._cached_matrix = self._compute_matrix(params, self._last_input_shape)
        return self._cached_matrix

class _InvertibleMixin:
    invertible: bool = True
    def _inverse_apply(self, x: torch.Tensor, params: dict) -> torch.Tensor:
        raise NotImplementedError
    def inverse(self, x: torch.Tensor, params=None) -> torch.Tensor:
        params = params or self._last_params
        if params is None:
            raise RuntimeError(
                "inverse requires params; call with return_params=True or pass explicitly"
            )
        return self._inverse_apply(x, params)

class _MultiTargetMixin:
    """Marker — _dispatch already handles registry lookup for multi-input calls."""
```

A simple `Normalize` subclasses just `Transform`. `RandomHorizontalFlip` subclasses `Transform, _RandomMixin, _RigidMixin, _InvertibleMixin, _MultiTargetMixin`. Each mixin adds ~10-20 lines of opt-in code.

### 4.3 Tagged tensor types — multi-target dispatch

```python
# kornia/tensors/__init__.py
class Image(torch.Tensor): ...

class BoundingBoxes(torch.Tensor):
    image_size: tuple[int, int]
    format: str  # "xyxy" | "xywh" | "cxcywh"

class Mask(torch.Tensor): ...

class Keypoints(torch.Tensor):
    image_size: tuple[int, int]

class Video(torch.Tensor): ...
```

Multi-target dispatch is **type-driven**:

```python
# Image-only — only Image kernel runs
img_out = aug(img_tensor)

# Multi-target — Image + BoundingBoxes kernels run with shared params
img_out, boxes_out = aug(K.tensors.Image(img), K.tensors.BoundingBoxes(boxes, ...))

# Legacy data_keys= still works (deprecated, removed in 3.0)
img_out, boxes_out = aug(img, boxes, data_keys=["input", "bbox"])  # warns
```

**Image-only calls pay zero multi-target dispatch overhead.** This is the #1 architectural win — most calls are image-only.

### 4.4 Per-sample randomness — opt-in via constructor

```python
class RandomHorizontalFlip(Transform, _RandomMixin, _RigidMixin, _InvertibleMixin, _MultiTargetMixin):
    def __init__(self, p: float = 0.5, *, per_sample: bool = False):
        super().__init__()
        self.p = p
        self.per_sample = per_sample    # default False — matches torchvision
```

**Default `per_sample=False`** flips kornia's historic default to match torchvision's batch-shared semantics. Users who need per-sample variation (SSL, DINOv2-style multi-view, contrastive recipes) set `per_sample=True` explicitly.

`presets.dinov2`, `presets.simclr`, `presets.mocov3` set `per_sample=True` automatically. Users following recipes don't have to think about it.

### 4.5 Lazy metadata

`_params` dict and `_transform_matrix` tensor are not constructed unless requested:

```python
out = aug(x)                         # hot path: zero metadata cost
out, params = aug(x, return_params=True)  # opt-in: constructs params dict
matrix = aug.transform_matrix        # computed on first read, cached
```

The hot path saves:
- `aten::full` for `batch_prob` (~0.6 ms self CPU per call from profile)
- `aten::tensor` + `aten::expand` for `transform_matrix` (~0.5-2 ms per call)
- dict construction for `_params` (~0.1 ms but high alloc count)

Total: ~1-3 ms saved per call, validated by the profile data.

### 4.6 Functional API mirror

```python
import kornia.augmentations.functional as KF

out = KF.random_horizontal_flip(x, p=0.5, generator=g)
out = KF.affine(x, degrees=15.0, translate=(0.1, 0.1), scale=(0.8, 1.2), generator=g)
out = KF.normalize(x, mean=KF.IMAGENET_MEAN, std=KF.IMAGENET_STD)
```

Bare functions, no class overhead. For users who want torchvision-functional-style direct kernel calls. Goal: parity with `torchvision.transforms.v2.functional`.

### 4.7 Concrete transform — RandomHorizontalFlip end-to-end

```python
# kornia/augmentations2/_2d/geometric/horizontal_flip.py
from kornia.augmentations2.base import (
    Transform, _RandomMixin, _RigidMixin, _InvertibleMixin, _MultiTargetMixin,
    register_kernel,
)
from kornia.tensors import Image, BoundingBoxes, Mask, Keypoints
import torch


class RandomHorizontalFlip(Transform, _RandomMixin, _RigidMixin, _InvertibleMixin, _MultiTargetMixin):
    """Random horizontal flip — fast-default with optional per_sample.

    Examples:
        >>> aug = RandomHorizontalFlip(p=0.5)             # batch-shared, fast
        >>> out = aug(image)                               # hot path
        >>> aug2 = RandomHorizontalFlip(p=0.5, per_sample=True)  # SSL-style
        >>> out_ssl = aug2(image)
    """

    def __init__(self, p: float = 0.5, *, per_sample: bool = False) -> None:
        super().__init__()
        self.p = p
        self.per_sample = per_sample

    def _sample_params(self, shape, generator=None):
        if self.per_sample:
            return {"flip": torch.rand(shape[0], generator=generator) < self.p}
        return {"flip": torch.rand(1, generator=generator).item() < self.p}

    def _apply(self, x, params):
        flip = params["flip"]
        if isinstance(flip, bool):
            return x.flip(-1) if flip else x
        return torch.where(flip[:, None, None, None], x.flip(-1), x)

    def _compute_matrix(self, params, shape):
        flip = params["flip"]
        W = shape[-1]
        if isinstance(flip, bool):
            if not flip:
                return torch.eye(3).unsqueeze(0)
            return torch.tensor([[[-1, 0, W-1], [0, 1, 0], [0, 0, 1]]], dtype=torch.float32)
        # per-sample matrix construction
        eye = torch.eye(3, dtype=torch.float32).unsqueeze(0).expand(flip.shape[0], 3, 3).clone()
        flip_mat = eye.clone()
        flip_mat[:, 0, 0] = -1
        flip_mat[:, 0, 2] = W - 1
        return torch.where(flip[:, None, None], flip_mat, eye)

    def _inverse_apply(self, x, params):
        return self._apply(x, params)  # involution


@register_kernel(RandomHorizontalFlip, Image)
def _hflip_image(t, x, params):
    return t._apply(x, params)


@register_kernel(RandomHorizontalFlip, BoundingBoxes)
def _hflip_boxes(t, boxes, params):
    flip = params["flip"]
    W = boxes.image_size[-1]
    flipped = boxes.clone()
    flipped[..., [0, 2]] = W - boxes[..., [2, 0]]
    if isinstance(flip, bool):
        return flipped if flip else boxes
    return torch.where(flip[:, None, None], flipped, boxes)


@register_kernel(RandomHorizontalFlip, Mask)
def _hflip_mask(t, m, params):
    flip = params["flip"]
    if isinstance(flip, bool):
        return m.flip(-1) if flip else m
    return torch.where(flip[:, None, None, None], m.flip(-1), m)


@register_kernel(RandomHorizontalFlip, Keypoints)
def _hflip_keypoints(t, kp, params):
    flip = params["flip"]
    W = kp.image_size[-1]
    flipped = kp.clone()
    flipped[..., 0] = W - kp[..., 0]
    if isinstance(flip, bool):
        return flipped if flip else kp
    return torch.where(flip[:, None, None], flipped, kp)
```

**~75 lines for a transform that supports image, boxes, masks, keypoints, inverse, transform_matrix, per-sample, replay.** Current kornia `RandomHorizontalFlip` is ~165 lines and covers strictly less.

## 5. Backwards compatibility

### 5.1 Two namespaces during migration

```python
import kornia.augmentation as K       # 1.x — exact current behavior, unchanged
import kornia.augmentations as K2     # 2.0 — new fast path
```

The `kornia.augmentations` namespace currently re-exports from `kornia.augmentation` (the 1.x funnel work). 2.0 replaces those re-exports transform-by-transform with new implementations.

Existing code keeps working. New code opts into 2.0 by importing the plural namespace.

### 5.2 BC contract preserved

All items from RFC 1.0 §4 (BC contract) remain enforced:
- All ~80 public class names in `kornia.augmentation.*`
- All submodule import paths
- The 7 rf-detr-critical class signatures (frozen-signature test)
- All `data_keys=` literal strings (`"input"`, `"image"`, `"mask"`, `"bbox"`, `"bbox_xyxy"`, `"bbox_xywh"`, `"keypoints"`, `"class"`, `"label"`)
- `kornia.geometry.boxes.{Boxes, VideoBoxes}`, `kornia.geometry.keypoints.{Keypoints, VideoKeypoints}`
- `transform_matrix` property on rigid transforms
- 4,832 existing tests + 194 numerical goldens

### 5.3 Deprecation timeline

- **2.0.0** (~10 weeks from approval): new namespace ships with ~20 most-used transforms migrated to fast-default. Old works unchanged.
- **2.1.0** (3 months later): full transform migration. `same_on_batch` deprecation warning at construct time when used.
- **2.2.0** (6 months after 2.0): `data_keys=` deprecation warning at call time.
- **3.0.0** (1 year after 2.0): `kornia.augmentation` (singular) becomes alias of `kornia.augmentations` (plural). Deprecations enforced.

### 6.2 DETR-style pipeline

```
Pipeline: HFlip + Affine + ColorJitter + Normalize, B=8, 512×512

Current kornia 1.x:    58.1 ms
torchvision.v2 eager:  24.3 ms

Albumentations CPU + 8 workers:  4.1 ms
```

### 6.3 Why per-sample remains 2-3× of torchvision

torchvision applies one decision per call, broadcasting. kornia per-sample applies B different transformations:
- B different rotation angles → B different (3,3) matrices → B different grids → B different `grid_sample` warps
- One angle → 1 matrix → 1 grid → 1 warp (B× broadcast)

These are different computations. Per-sample is structurally more work. The pitch: "per-sample mode delivers DINOv2-style multi-view in a single batch — torchvision can't do this without N separate forward calls."

## 7. Implementation phases

### Phase 1 — Foundation (3 weeks)

- New base class + 4 mixins under `kornia/augmentations2/base.py`
- Tagged tensor types (`kornia.tensors.{Image, BoundingBoxes, Mask, Keypoints, Video}`)
- Type-dispatch kernel registry
- Lazy metadata pattern with property-based `transform_matrix`
- Functional API mirror (`kornia.augmentations.functional`)
- 5 exemplar transforms migrated: HFlip, Normalize, CenterCrop, RandomAffine, ColorJiggle
- Bench on Jetson Orin + x86 CI runner — establish kornia 2.0 baseline numbers
- All goldens green for the 5 exemplars (regenerated where per_sample default flip changes RNG sequence)

**Acceptance:** 5 transforms benchmark within projection (HFlip ≤ 2 ms, CenterCrop ≤ 0.5 ms, Normalize ≤ 3 ms), full test parity, RFC sign-off from maintainers.

### Phase 2 — Mass migration (4 weeks)

- ~80 transforms migrated to new bases. Each PR: 5-10 transforms + their tests + their goldens.
- Old classes preserved as deprecation shims at the singular namespace.
- `per_sample` default flipped per transform. Migration guide documents the change for callers who relied on `same_on_batch=False` (now opt-in via `per_sample=True`).
- Goldens regenerated where semantics differ (per_sample flip changes RNG draw sequence).

**Acceptance:** all 80 transforms in 2.0 namespace, all goldens green or explicitly regenerated with rationale, downstream rf-detr/torchgeo canary tests still pass.

### Phase 3 — Multi-target + presets (2 weeks)

- Multi-target dispatch via tagged tensors validated end-to-end
- `BboxParams(format, min_area, min_visibility, label_fields)` ported from 1.x funnel
- Mask handling via tagged `Mask` tensor (no fake channel-dim requirement — closes torchgeo concern T7)
- Keypoint formats
- Recipe presets:
  - `presets.detr(resolution=560)` — DETR detection training
  - `presets.dinov2(local_views=8)` — SSL multi-view, `per_sample=True`
  - `presets.simclr`, `presets.mocov3`
  - `presets.yolov8()` — Mosaic 2×2 + RandomAffine + HSV + flip
  - `presets.sam()` — Segment Anything pipeline
- Compat importers: `compat.from_albumentations(spec)`, `compat.from_torchvision_v2(compose)`

**Acceptance:** every preset reproduces published recipe numbers within ε, compat importers emit fidelity reports for unsupported transforms, end-to-end DINOv2 training step passes without divergence.

### Phase 4 — Validation (1 week)

- Bench leaderboard on A100, 4090, Jetson Orin
- Reproduce DETR mAP / DINOv2 linear-probe / YOLOv8 mAP within ε of published numbers
- Documentation, migration guide, RFC closure
- Release candidate

**Total: 10 weeks for kornia 2.0.**

## 9. Risks and open questions

### 9.1 Risks

- **R1 — Goldens drift on per_sample default flip.** Migrating callers from `same_on_batch=False` (current default) to `per_sample=False` (2.0 default) changes RNG draw sequences. Goldens regenerated transform-by-transform with explicit reviewer sign-off per PR.
- **R2 — TorchScript users.** Pydantic params and the type-dispatch registry don't script. Documented as a non-goal; users needing scripted preprocessing use the functional API.
- **R3 — Mixin debugging complexity.** Composing 4 mixins is harder to debug than 5-level inheritance for a maintainer learning the codebase. Mitigated by clear mixin documentation + small mixin sizes (10-20 LOC each).
- **R4 — Cross-repo coordination.** torchgeo, rf-detr, lightly all depend on `kornia.augmentation` (singular). Migration to plural is opt-in for downstream; we don't force them. Coordination plan: ping maintainers when 2.0 lands; offer migration help.
- **R5 — Pydantic v2 dependency floor.** kornia 1.x has zero non-PyTorch deps. Pydantic v2 is the first. Acceptable cost; users can install `kornia[no-pydantic]` for a stripped subset. Internal validation falls back to plain dataclass when pydantic unavailable.

### 9.2 Open questions for maintainer review

- **Q1.** `data_keys=` removal in 3.0 — confirm 1-year deprecation window is the right horizon.
- **Q2.** Should `presets.*` be a separate package (`kornia-presets`) to avoid bloating core, or stay in-tree? Recommendation: in-tree for discoverability; separate PyPI later if size becomes an issue.
- **Q3.** Tagged tensor types — should they live in `kornia.tensors` (new top-level) or `kornia.augmentations.tensors`? Recommendation: top-level, since boxes/keypoints are useful outside augmentation contexts.
- **Q4.** Compat importer scope — Albumentations has ~80 transforms. Should we translate all or freeze at top-25 by usage? Recommendation: freeze at 25, raise `NotTranslatable` with suggestion for others.
- **Q5.** Should the kornia 2.0 release ship as kornia 1.x.0 (minor) or 2.0.0 (major)? Recommendation: 2.0.0 — the architectural change warrants the major bump even though most public APIs are preserved via re-exports.
- **Q6.** MedianBlur kernel rewrite — should this be its own RFC or rolled into 2.0? Recommendation: separate fix-up PR before 2.0 ships, since it's purely a kernel change with no architecture implications.

- `benchmarks/profile/bottlenecks.json` — torch.profiler per-op bottleneck data on 8 worst offenders
- `benchmarks/profile/bottlenecks_all.json` — full-sweep profile (all 37 transforms, in progress)

The empirical pattern is consistent: kornia 1.x pays ~3-5 ms of framework overhead per call on Jetson Orin GPU, which dominates kernel time for cheap ops. The 2.0 redesign targets this overhead specifically while preserving all kornia-unique features as opt-in.

## Appendix B — What 2.0 explicitly does NOT change

To preserve scope and keep the migration tractable:

- `kornia.geometry.*` — untouched
- `kornia.color.*` — untouched
- `kornia.feature.*` — untouched (gradcheck tests there are intentional, kornia.feature is differentiable)
- `kornia.losses.*` — untouched
- `kornia.morphology.*` — untouched
- `kornia.contrib.*` — untouched
- `kornia.onnx.*` — untouched
- `kornia.models.*` — untouched

Only `kornia.augmentation.*` (and the new `kornia.augmentations.*`) are in scope.
